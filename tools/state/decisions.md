# Durable decisions

- **2026-07-13 — Require decision-grade identity before comparing agent runs.** Compact v2 rows retain task version, resolved repository commit, Codex CLI, prompt/handoff/inspection modes, P2P status, route, gates, cost, failure provenance, and artifact pointer. `tools/task-metrics.py compare` refuses unequal task sets/counts or missing/mismatched task versions, then reports repeated portfolio distributions and promotion gates. Raw trajectories remain out-of-band behind pointers so diagnostic depth does not become default context cost.
- **2026-07-13 — Use focused inspection only for local edits in large files.** Give the worker a task-specific `rg` literal, cap inspection at 40 surrounding lines, avoid generic syntax searches/whole-file output, and require the playbook's preservation method. Across three two-task portfolios, all six candidate tasks scored 100 while median effective tokens fell 21.7% and tool output 59.2%; total duration rose 5.2%. A weaker “bounded” instruction caused a CRLF scope failure and is rejected. Broad refactors/diagnosis remain exempt.
- **2026-07-13 — Promote process-wide token optimizations from repeated portfolios, not single runs.** Use at least two representative tasks and three matched baseline/candidate portfolio samples; require every capability gate, at least 5% lower median effective tokens, and no unexplained failure/tool-output/duration regression. Treat overlapping token ranges as inconclusive, then run the full visible suite plus one held-out task after candidate freeze. Runner-lite's three-sample range (29,098–30,046) did not overlap durable handoff (42,958–57,936), with equal 100/100 capability.
- **2026-07-13 — Use schema-free runner capture for short instrumented tasks.** When one runner can durably retain raw JSONL, stderr, patch, independent grade, metrics, and final response, the worker may skip its Markdown report, log append, and duplicated exhaustive checks. Keep output-file-first for lookup, long, incremental, or interruption-prone tasks. In matched WBD-001/003 trials, runner-lite kept both scores at 100 while reducing effective tokens 15.2%, tool output 12.6%, and worker duration 56.7%; enforcing a redundant final-response schema increased effective tokens 47.8% and is reserved for actual response consumers.
- **2026-07-13 — Route bounded explicit JS/visual repairs to Terra/low, and audit repeated failures before further escalation.** Privacy JS and reference-image CSS each scored 100 at Terra/low, saving 39.1% and 19.7% effective tokens versus Terra/medium. If two routes miss the same deterministic assertion, inspect the task contract/grader for hidden or representation-specific requirements before a third model spend; version corrections and restart from the cheapest plausible route.
- **2026-07-13 — Use progressive disclosure for Codex work.** Keep auto-loaded `AGENTS.md` concise, do not ask workers to print it back, skip unrelated global ledger reads for self-contained dispatches, name only domain-relevant playbooks/source, and precreate ignored handoff directories. A four-task matched run improved capability 2/4 to 3/4 while cutting effective tokens 8.2%, input 25.5%, worker time 21.8%, and tool output 36.7%; both previously passing tasks stayed 100/100 with lower effective tokens.
- **2026-07-13 — Token optimization is capability-gated by a fixed local web-development suite.** Use detached real-repository tasks with hidden F2P assertions, existing P2P checks, deterministic browser evidence, actual runner token telemetry, and one held-out multi-file task. Retain an optimization only when every critical assertion passes, every task scores at least 85/100, the mean does not fall below baseline, and matched passing runs reduce total worker plus review/repair cost. Rationale: upstream suites supply useful methods but do not match this bilingual build-free maintenance workload closely enough to import wholesale.
