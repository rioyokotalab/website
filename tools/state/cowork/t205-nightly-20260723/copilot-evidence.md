## Critique

1. **Question 1, resolved — both an invocation error and an allowlist boundary, but not the way the independent evidence framed it.** The independent evidence guessed the `--selftest` flag typo mattered. It does not: this session ran the *correct* `selftest`/`audit` subcommand syntax and both were still denied, with the same generic message as before. So the earlier claim "denial was because of the allowlist" is correct in outcome but was reached from an untested premise (that fixing the flag would change anything). **Accept**: python3 execution of any form is currently out of the co-pilot's tool boundary in this session. **Reject**: any inference that a corrected-syntax call would have behaved differently — it did not. Practically, this means neither this session nor the imported one has ever executed `selftest`/`audit`; only the driver has (driver-evidence.md §Commands 1-2). The frozen protocol should not assume the co-pilot can independently reverify selftest/audit pass/fail state — that check is driver-only unless the reviewed allowlist is widened to include `python3 tools/agent-benchmark/benchmark.py {selftest,audit}` specifically.
2. **Driver's WBD-001 matched probe (question 2).** Accept as capability evidence: 100/100 on both the current run and the matched historical cell, same two files changed, is real signal that the current GPT-5.6-terra/low-effort/WBD-001 path still works end-to-end. Reject the framing of the 10.430-second delta and +118-token delta as anything beyond noise — the driver's own text already flags this ("one pair cannot establish a trend"), and I concur: with n=1 per side, direction and magnitude of any runtime/token change are unresolved until matched repeats exist. No held-out or other-task capability claim should be drawn from this single WBD-001/low/gpt-5.6-terra cell.
3. **Run ID vs. run label (question 3), resolved.** `run_matrix.py:45` shows `run_id` is already a pure deterministic function of `(label, task, model, effort)` — it does not and structurally cannot encode `prompt_mode`/`handoff_mode`/`inspection_mode`, which only exist in `benchmark.py`'s `run_claude()` signature (line 388-390), a different layer. Two designs were on the table: (a) fold workflow modes into `run_id` composition, or (b) require a unique run label per mode combination and keep `run_id` as-is. Given the plan's own structure — a single frozen protocol governs the broad singleton matrix (step 6), while mode variation is confined to the bounded iteration step (step 8) on visible tasks only — encoding modes into every `run_id` would bloat the identifier for the common case (one frozen mode set) to guard a risk that only exists during iteration. **Resolution: require a unique `run_label` per distinct mode combination** (e.g. `postfreeze-2026-07-23-runnerlite-default` vs. `...-compact-terse`), record the frozen `prompt_mode`/`handoff_mode`/`inspection_mode` as metadata tied to that label, and add a `run_matrix.py` selftest assertion that no two `results.jsonl` rows share a `run_id` (already enforced by `read_results()`) while also asserting no two distinct mode-metadata sets share a `run_label`. This is lower-risk than changing the `run_id` schema mid-benchmark and keeps the broad matrix's identifiers stable.
4. **Time budget (question 7).** The only concrete current-session timing evidence is the driver's single low-effort WBD-001 probe: 55.664s total / 37.311s worker now vs. 66.094s / 49.519s historically. This is one cell at the cheapest effort tier; per-cell cost at `xhigh`/`max`/`ultra` is unmeasured and must not be extrapolated from it (the independent evidence's "~2.65 min/cell average" figure was an unlabeled arithmetic artifact of 435÷165, not a runtime estimate, and I retract it rather than restate it as budget guidance). Instead of inventing throughput, propose a **checkpoint-and-truncate rule**: run cells in the plan's existing priority order (visible tasks and capability probes before held-out, GPT then Claude alternating to keep both providers' partial coverage comparable if truncation occurs); after every 15 completed cells, compute the rolling mean wall-clock per cell so far and project remaining time; if projected finish exceeds (8h − 45min reserve), stop starting new *repeat* cells first, then stop new *held-out* cells, then stop new *singleton* cells in reverse-priority order, always let an in-flight cell finish rather than killing it. This uses only measured throughput, recomputed live, rather than a single a-priori number.
5. **Narrowest Claude invocation + denial-path probe (question 5), addressed without invoking a model.** Charter §Baseline already specifies the target shape: `claude -p --output-format stream-json --verbose --model <model> --effort <effort> --permission-mode dontAsk --allowedTools <reviewed list>`, dropping `--dangerously-skip-permissions` entirely (confirmed absent from current code, Commands §5). The reviewed tool list should be the minimum the five task definitions actually need — file read/edit within the capsule workspace, a local server start/stop command, and a screenshot tool — and explicitly exclude `git push`, `ssh`, `lftp`, publish/deploy scripts, and unscoped `Bash(*)`. For the denial-path probe: this session itself is a live, already-executed instance of that exact failure mode — a `dontAsk`-mode denial on an out-of-allowlist command (`python3 ...`) returned **immediately** with a clear, structured denial message; it did not hang or time out. That directly answers the open question in the independent evidence ("does a denied tool call abort with a clear artifact, or hang?") for Bash-tool-level denials: it aborts cleanly. The still-unresolved part is whether `benchmark.py`'s `run_claude()` subprocess wrapper (stdout/stderr capture, timeout handling at `timeout_seconds`) surfaces that same clean denial as a scored failure in `result.json` rather than as a hang inside the subprocess — that requires running an actual Claude worker call under the new flags with a fixture-scoped denial trigger, which is out of scope here (no model invocation) and should be the "standalone gate before Step 5" item already proposed.

## Proposed plan changes

1. Do not add a plan step assuming the co-pilot can independently rerun `selftest`/`audit`; either widen the reviewed allowlist to permit exactly `python3 tools/agent-benchmark/benchmark.py {selftest,audit}` (no other python3 forms) if independent co-pilot reverification is wanted, or explicitly assign that check to the driver only and record co-pilot verification as inspection-of-driver-artifact, not independent execution.
2. Freeze the run-identity design as: `run_id` stays `(label, task, model, effort)`-only per the existing `run_matrix.py:45` function; require a unique `run_label` per distinct `(prompt_mode, handoff_mode, inspection_mode)` combination; add a selftest assertion that rejects two `results.jsonl` rows sharing a `run_label` with different recorded mode metadata.
3. Replace any single-number time-budget estimate with the checkpoint-and-truncate rule in Critique §4 (rolling mean every 15 cells, tiered truncation: repeats → held-out → singletons, always let in-flight cells finish), computed from cells actually run this session, not from the 8h/165 arithmetic mean.
4. Before Step 5 (per driver evidence item 3 / independent evidence item 2, both of which I accept and merge into one gate): require the Claude invocation to drop `--dangerously-skip-permissions` and instead use `--permission-mode dontAsk` with a named, reviewed `--allowedTools` list scoped to capsule file edit + local server + screenshot only; require one WBD-001 visible probe under that invocation that (a) scores normally when no denial occurs, and (b) is separately tested against one deliberately out-of-scope command to confirm `run_claude()`'s subprocess handling surfaces a clean scored failure rather than a hang — this second check is still open and must run before any broad Claude matrix.
5. Retract the independent evidence's "~2.65 min/cell" figure from the frozen protocol; it was unlabeled arithmetic, not a measurement, and conflicts with the checkpoint rule above.
6. Everything else in the imported independent evidence not revised above — the `ARTIFACTS`/`FREEZE` path-reconciliation finding (run_matrix.py:20 vs. benchmark.py's `tools/agent-benchmark/artifacts`), the missing Claude freeze file, and the provider-specific (not shared) repeat-count/stop-rule recommendation — stands and is **accepted** as still correct after this session's rereading of the same source lines.
