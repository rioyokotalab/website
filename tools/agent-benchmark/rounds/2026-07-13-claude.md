# Claude agentic-harness benchmark round — 2026-07-13

## Frozen question

Compare three isolated Claude configurations on the frozen WBD-001–005
capability suite, capability before efficiency, with everything else pinned:
repository/config ref `3364e2c3617b1fa0d0d044a8d5a5d1af3faa548d`, model `opus`,
effort `high`, CLI `2.1.207`, arm-neutral prompts, task/grader hashes, timeout
class, mutation, authorized paths, and P2P/visual gates. Each measured run is a
fresh, non-persistent session in a throwaway Git fixture with publish/deploy and
remotes removed. Protocol: `skills/claude-benchmark.md`.

## Arm identities

- **current-harness** — project config frozen at the ref (CLAUDE.md, site
  agents, routing, Codex MCP offload); 11 config files, 62,497 instruction
  bytes; config sha256 `502bb480…`; entry agent `site-coordinator`, lifecycle
  hooks disabled.

- **dynamic** — clean start, then a tool-free safe-mode planning call generates

  executor. Generation counts against the arm's budget.

## Rejected / generated strategies

- **Dynamic generation reliability defect (stochastic, not schema-incompatible).**
  The dynamic config generator failed with `error_max_structured_output_retries`
  on 3 of 8 calls: WBD-003 ×1 and WBD-005 ×2. Diagnosis from telemetry:
  successful generations finish in 2 turns (~1.8–4.9k output tokens); every
  failure spirals to 6 turns (~9–12k tokens) and exceeds the schema `maxLength`
  caps. The same frozen command therefore sometimes succeeds and sometimes
  fails — WBD-003 recovered on an unchanged retry, and WBD-005 recovered on its
  third unchanged attempt (2-turn success). Excluded run IDs (in `exclusions.json`,
  relabeled `claude-dynamic-screen-v1-excluded-setup`): `…-wbd-003-eacbd2`,
  `…-wbd-005-6a4f2b`, `…-wbd-005-c2eacd`. Failed-generator spend $1.1283 is
  attributed to the dynamic arm. Each dynamic task is measured on its first
  successful generation — the identical rule applied to all five tasks; no
  schema/prompt/profile/route/runner invariant was changed to obtain the WBD-005
  result.
- **Dynamic's per-task generation produced the decisive WBD-005 win.** On the
  hardest held-out task the generated config plus a fresh executor passed 100
  where the empty autonomous config (87) and the heavy current harness (78)
  both failed. On the visible set all arms already scored 100, so the advantage
  is concentrated on the complex multi-file task.

## Recommendation

- **Dynamic is the sole arm to pass the full 5/5 suite** and clears the
  capability gate over current-harness, but it is a *per-task generation
  strategy*, not a drop-in canonical config; its generator has a stochastic
  structured-output reliability defect (3/8 calls failed); and this is a
  one-run-per-task screen, not a variance estimate. **Do not wholesale-replace
  the canonical configuration on a single screen.** Keep `current-harness`
  canonical for now.
- **Pursue the task-specific-generation approach as the most promising
  direction**, gated on: (1) hardening generator reliability (retry/backoff,
  bounded output, or a simpler/again-validated structured schema) so it does
  not waste budget or block a task; and (2) a confirmatory repeat phase
  (representative tasks in matched portfolios) before any adoption.
- **Autonomous** remains a strong efficiency signal: it matched visible-task
  capability at −74% Claude-side tokens vs current-harness, and dynamic's 5/5
  costs +186% effective tokens over autonomous — so the capability/cost
  tradeoff, not raw tokens, is the real decision axis for a future phase.
- WBD-005 stays the discriminating task: reduced-motion / query-version
  behavior (`standards-check.py`, `js-reduced-zero`) is where the weaker arms
  break.
