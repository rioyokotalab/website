## Frozen question

capability suite, capability before efficiency, with everything else pinned:
repository/config ref `3364e2c3617b1fa0d0d044a8d5a5d1af3faa548d`, model `opus`,

## Recommendation

- **Keep `current-harness` as the canonical configuration.** No arm passed the
  full 5/5 suite, so the protocol licenses no capability-gated winner and no
  config change is adopted from this screen.
- Record two durable signals for future work: (1) the **autonomous** minimal
  configuration matches visible-task capability at far lower Claude-side tokens
  and cost — a candidate worth a dedicated 5/5-targeted phase, not a screen;
  (2) the **dynamic** generation path has a structured-output reliability defect
  that must be hardened (retry/backoff or schema simplification) before it is a
  viable arm — it cost budget without producing a WBD-005 result.
- WBD-005 remains a discriminating task: every arm that reached execution missed
  it, all on the `standards-check.py` P2P gate (current-harness also on
  `js-reduced-zero`). Reduced-motion/query-version behavior is the weak spot.
